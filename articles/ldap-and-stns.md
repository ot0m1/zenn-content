## はじめに：なぜこの記事を書くか

「STNSって、LDAPを置き換えたやつでしょ」。私のSTNSへの理解は、長らくその程度でした。名前は知っている、なんとなく軽いらしい、でも中で何が起きているのかは説明できない。この記事は、その状態から「人に説明できる」ところまで、自分の理解を引き上げるために書いています。

STNSをちゃんと理解するには、遠回りに見えても、まずLDAPを理解するのが近道です。STNSは「LDAPの何かを、別のやり方で置き換えたもの」なので、置き換えられた側（LDAP）が分からないと、STNSの何がうれしいのかもピンと来ないからです。だからこの記事は、LDAPから入ります。

ちなみに、タイトルの「LDAPとSTNS」という並べ方には、実は小さな食い違いがあります。LDAPはプロトコルの名前で、STNSはOSS製品の名前——つまり、厳密には同じ種類のものを並べていません。この食い違いを整理するところが、そのまま理解の入口になります（§3で詳しく扱います）。

この記事を最後まで読むと、次のことが一本の線でつながります。

- なぜ、そもそも「中央でユーザーを管理するしくみ」が必要なのか
- LDAPはどう動いていて、なぜ「重い」と言われるのか
- STNSはその何を「簡単」にしたのか。そして、どうやって実現しているのか
- そして、業界の次はどこへ向かおうとしているのか

それでは、いちばん素朴なところ——サーバーが1台だけの世界——から始めます。

---

## 1. そもそも何を解決したいのか — 中央ユーザー管理

まず、LDAPもSTNSも「同じ一つの問題」を解くための道具だ、というところから始めます。その問題とは、たくさんのサーバーにまたがってユーザーをどう管理するか、です。

サーバーが1台しかないうちは、悩む必要はありません。Linuxのユーザーは `/etc/passwd`・`/etc/shadow`・`/etc/group` といったファイルに直接書いてあり、アカウントを足したければそのファイルをいじればいい。それで完結します。

ところが、サーバーが100台に増えたらどうなるでしょうか。入社した人を1人追加するために、100台ぶんのファイルを編集する。退職した人を消すために、また100台を触る。1台でも消し忘れれば、その人はまだそのサーバーに入れてしまう。どのサーバーが正しいアカウント一覧を持っているのか、誰にもわからなくなります。「唯一の正しい情報源（single source of truth）」が存在しない状態です。

その問題の回答として次のようなアプローチがあります。ユーザー情報を中央の1箇所に集め、各サーバーは自分でアカウントを抱えず、必要なときにその中央へ問い合わせる。ログインのたびに「この人は誰で、入っていいのか」を中央に聞きにいく形にすれば、追加も削除も中央の1箇所を直すだけで済みます。

LDAPも、この記事で扱うSTNSも、やっていることの根っこはこれです。「中央にユーザーを置いて、各サーバーはそこに問い合わせる」——この同じ発想を、違うやり方で実装したもの同士。だからこそ、片方（LDAP系のしくみ）を理解すれば、もう片方（STNS）は「どこを変えたのか」だけで理解できます。

## 2. その前に：Linuxはどこで「ユーザー」を解決しているか — NSSとPAM

LDAPやSTNSの中身に入る前に、どうしても先に押さえておきたい土台があります。そもそもLinuxは、ログインしようとしている「ユーザー」を、どこで・どうやって確かめているのか、という話です。ここがわかっていないと、LDAPもSTNSも「結局どうやってユーザーを管理しているのか」がぼやけてしまいます。

Linuxがユーザーを扱うとき、実は性質の違う2つの問いが発生しています。

1つめは「この人は誰か」。ユーザー名 `alice` に対して、uid はいくつか、所属グループ（gid）は何か、ホームディレクトリやログインシェルはどこか——といった、身元と属性を調べる問いです。これを担当するのが **NSS（Name Service Switch）** で、設定ファイルは `/etc/nsswitch.conf` です。

2つめは「この人を通していいか」。入力されたパスワードは正しいか、そもそもこの人はこのサーバーにログインしてよいのか——という、認証の問いです。これを担当するのが **PAM（Pluggable Authentication Modules）** で、設定は `/etc/pam.d/` 以下に置かれます。

「誰か（NSS）」と「通していいか（PAM）」は、似ているようで別の問いです。ここを分けて意識できると、この先がぐっと楽になります。

そして、この2つに共通する最大の特徴が、どちらも**プラグイン式**だということです。名前のなかの「Switch」も「Pluggable」も、まさにそれを指しています。標準では、どちらもローカルのファイル（`/etc/passwd` など）を見にいくように設定されています。`/etc/nsswitch.conf` にはこう書かれています。

```
passwd:  files
group:   files
shadow:  files
```

この `files` は「ローカルファイルを見る」という意味のプラグイン指定です。ここを差し替えれば、Linuxはユーザー情報を別の場所から取ってくるようになります。たとえば、

```
passwd:  files ldap
```

と書けば、「まずローカルファイルを見て、いなければLDAPに問い合わせる」という動きになります。`ldap` の部分を別のものに変えれば、また別の情報源になります。

この「差し替えられる」という一点こそが、この記事全体の土台です。LDAPもSTNSも、**NSSとPAMがもともと用意している「差し込み口」に、自分用のプラグインを挿している**だけです。

つまり、これ以降の話はすべて、「その差し込み口に、何を・どうやって挿すか」の違いにすぎません。LDAP系のしくみはLDAPを喋るプラグインを挿し、STNSはHTTPを喋るプラグインを挿す。土台となるNSSとPAMは同じです。次の章からは、まずLDAPがそこに何を挿しているのかを見ていきます。

## 3. LDAPとは何か — そして「LDAPとSTNS」は本当は土俵違い

「LDAPとSTNS」と並べると、つい同じ種類のものを比べている気になります。どっちも英語4文字ですし。でも、この2つはそもそも同じカテゴリの言葉ではありません。

**LDAPはプロトコルの名前です。** Lightweight Directory Access Protocol の略で、名前のとおり「ディレクトリ（サービス）にアクセスするための通信の作法」——RFCで決まった喋り方の規格であって、インストールして動かす製品ではありません。HTTPが製品ではなく「Webの喋り方」であるのと同じように、LDAPは「ディレクトリの喋り方」にあたります。

その喋り方でアクセスする相手が、ディレクトリサービスです。木構造で、読み出しに最適化された台帳のようなデータベースで、ユーザーやグループを階層に並べて格納します。

では実際にサーバーとして立てて運用するものは何かというと、そのディレクトリサービスを実装した製品のほうになります。代表的なのは次のものです。

- OpenLDAP … 定番のOSS実装
- 389 Directory Server … Red Hat系のOSS実装
- Active Directory … Microsoftの製品。LDAPに加えてKerberosやグループポリシーまで束ねた大きな塊で、「＝LDAP」ではありませんが、LDAPを喋ります

このあたりです。「インストールして、ユーザーを入れて、運用する物」はLDAPそのものではなく、これらの実装を指します。

ここまで来ると、最初の違和感の正体が見えてきます。STNSはOSS製品の名前です（次節以降で扱います）。一方のLDAPはプロトコルの名前。つまり「LDAPとSTNS」は、製品とプロトコルを並べていて、厳密には比較の土俵が半分ずれています。STNSと対等に比べたいなら、相手は「LDAP」ではなくLDAP系のサーバー実装——OpenLDAPあたり——です。「STNS vs OpenLDAP」で、はじめて同じ土俵に乗ります。この記事のタイトルは通りのいい「LDAPとSTNS」にしていますが、頭の中でこのズレだけ分けておくと、後がずっと読みやすくなります。

### LDAPは、ユーザー管理のために生まれたのではない

ここが、この記事で一番わかっておいてほしい点です。LDAP（正確にはその背後のディレクトリサービス）は、Unixのユーザーを管理するために作られた仕組みではありません。もっと大きい——「組織の参照情報を中央に一箇所へ集めて、みんなで引く」ための汎用の台帳です。ユーザー管理は、その台帳の使い道の一つにすぎません。

台帳に載るのはアカウントだけではありません。

- 人（社員、部署、役職、メール、電話番号）
- グループ
- 組織図そのもの（会社 → 部 → 課の階層）
- マシン、プリンタ、ネットワーク機器、サービス
- アプリの設定、証明書

共通するのは「しょっちゅう引くけれど、めったに書き換わらない、組織にまつわる参照情報」だという点です。木構造で、読み出しに最適化されているのは、まさにこの用途に振り切っているからです。逆に、注文履歴のように激しく更新されるデータは入れません——それは普通のデータベースの仕事です。

出自をたどると一発で腑に落ちます。LDAPの先祖は **X.500** という1980年代の規格で、狙いは「組織や人の情報を世界規模で引ける電子電話帳（ホワイトページ）」でした。ただしX.500は重厚なOSIの世界の代物で、アクセスも重いものでした。そこにTCP/IPで手軽にアクセスするために作られた軽量版の喋り方が、**L**ightweight **D**irectory **A**ccess **P**rotocol——LDAPです。名前の「Lightweight」は、X.500に対して軽い、という意味なのです。

つまりLDAPは、生まれからして住所録・ディレクトリの技術でした。そこへ後から、「組織の情報を中央に集められるなら、Unixのアカウント（uid・gid・ホーム・シェル・パスワード）を置くのにもちょうどいい」と、ユーザー管理という用途が乗っかりました。台帳の中身がアカウントとして使われるとき、その1件はこんな形をしています。

- 各エントリに DN（Distinguished Name＝識別名）。木のどこにいるかを示す住所のようなもの
- エントリは属性の集合。`uid` / `uidNumber` / `gidNumber` / `homeDirectory` / `loginShell` など
- `posixAccount` のような objectClass が「これはUnixアカウントだ」という型を与える

そして、汎用であるということは、それだけ作りが大きく、重いということでもあります。STNSは、この大きな台帳のうち「Unixのユーザーとグループ」という部分だけを切り出して単純に作り直したものです。なお、LDAPの「Lightweight」がX.500に対する軽さを指すのと同じく、STNSの「Simple」はLDAPに対する簡潔さを指しています。

## 4. LDAPの仕組み — ログインの一部始終

では、LDAP系のしくみ（ここではOpenLDAPを立てた構成を例にします）で、実際にログインがどう流れるのかを追ってみます。前章の「差し込み口」に、LDAP用のプラグインを挿すとどうなるか、という話です。

登場人物は2種類です。ひとつは中央に立てる **LDAPサーバー**（OpenLDAPなら `slapd` というデーモン）。ここに、木構造でユーザーが格納されています。もうひとつは、人が実際にログインしてくる **それぞれのサーバー**。こちらには、ユーザーは1人も登録しません。代わりに、

- NSS（`/etc/nsswitch.conf`）に `ldap` を足す
- PAM（`/etc/pam.d/`）に `pam_ldap` を足す

という設定をして、「アカウントは手元に持っていない。必要になったら中央のLDAPに聞け」という状態にしておきます。

なお、`nsswitch.conf` に書くのは「ldap という情報源を使う」という指定だけです。接続先のLDAPサーバーのアドレスや、木のどこから探すか（base DN）といった情報は、別のLDAPクライアント設定ファイル（`/etc/ldap.conf` など。近年よく使われるSSSD構成なら `/etc/sssd/sssd.conf`）に書きます。

この状態で `alice` がログインしようとすると、こう進みます。

1. `alice` がログインを試みる
2. 「`alice` って誰だ？」——NSSが **LDAPサーバーへ search（検索）** を投げ、uid・gid・ホームディレクトリ・ログインシェルを取ってくる（＝§2でいう「誰か」の解決）
3. 「このパスワードで通していいか？」——PAMが、`alice` のDNを使って **LDAPへ bind**（＝そのDNとパスワードでログインを試す操作）を行う。bindが成功すれば、パスワードは正しい（＝§2でいう「通していいか」の判定）
4. 両方パスすれば、`alice` はそのサーバーに入れます

ポイントは、ログインしてくる側のサーバーが完全に **手ぶら** だということです。自分ではアカウントを1件も持たず、ログインのたびに中央のLDAPへ「誰？」「通していい？」を問い合わせている。だから、ユーザーの追加も削除も、中央のLDAPを1箇所直すだけで全サーバーに反映されます。§1で欲しかった「中央の単一の情報源」が、これで実現できているわけです。

なお、ここで `search` と `bind` という2つの操作が出てきました。これがLDAP"プロトコル"の中身です。LDAPというプロトコルは、ざっくり言えば「木構造のディレクトリに対して bind／search／add／modify／delete する」ための取り決めだと思っておけば、この記事の範囲では十分です。

## 5. LDAPの何がつらいのか — STNSが生まれた理由

ここまで読むと、LDAPは中央管理の問題をきれいに解いているように見えます。実際、解いています。ただ、いざ実運用してみると、その代償として「重さ」がのしかかってきます。

第一に、データモデルが重い。前に見たとおり、LDAPの中身は木構造（DIT）で、各エントリには objectClass でスキーマが定められ、DNという住所の付け方にも作法があります。ユーザーを1人足すだけでも、この木のどこにどう置くか、どの objectClass を付けるか、を意識しなければなりません。データを流し込むのに LDIF という専用フォーマットも出てきます。要するに、覚えることが多い。

第二に、運用が難しい。中央のLDAPが1台では、そこが落ちると全サーバーがログインできなくなるので、レプリケーション（複製）と冗長化を組みます。誰がどのエントリを読み書きできるかのアクセス制御（ACL）も設計が要ります。`ldapsearch` や `ldapmodify` といったコマンドの独特な作法もあって、トラブルシュートも骨が折れます。学習コストが高いのです。

第三に——そしてこれが一番効いてくるのですが——多くの現場でやりたいことは、そこまで大げさではありません。「社内の各サーバーに、同じUnixユーザーとグループを配りたい」。ほしいのはそれだけなのに、汎用の巨大ディレクトリであるLDAPは、明らかにオーバースペックです。組織図もアドレス帳も証明書も入る汎用の台帳を、ユーザー配布のためだけに構築・運用することになります。

ここから出てくるのが「LDAPと同じこと（中央でユーザーを管理して各サーバーに配る）を、もっと単純にできないか」という発想です。その答えとして作られたのが、次に扱うSTNSです。

## 6. STNSとは何か / どう解決したか

そこで登場するのがSTNSです。**[STNS](https://github.com/STNS/STNS)＝Simple TOML Name Service**。[pyama86](https://github.com/pyama86) さんが作ったOSSで、名前がそのまま思想を表しています。「Simple（単純）」に、「TOML」で書く「Name Service（ネームサービス）」。

LDAPと同じ仕事——中央でユーザーを管理し、各サーバーに配る——をしながら、STNSは§5で挙げた重さを、次のように削ぎ落としています。

**設定がTOML。** 木構造もスキーマもありません。ユーザーの定義は、人が読んですぐ分かるTOMLで、こんなふうに書きます。

```toml
[users.alice]
id = 2001
group_id = 2001
directory = "/home/alice"
shell = "/bin/bash"
```

DITのどこに置くか、objectClass は何か、といった悩みが丸ごと消えています。ユーザーを足したければ、このTOMLに数行足すだけです。

**バックエンドがただのHTTP。** STNSサーバーは、このユーザー情報を **HTTPで返すだけ** のサーバーです。LDAPのような専用プロトコルはありません。「`alice` の情報をくれ」というHTTPリクエストに、JSONで答える。それだけです。専用プロトコルを覚える必要がなく、デバッグも `curl` で覗けます。

**クライアントは差し込み口に挿すプラグイン。** 各サーバー側には **libnss-stns** と **pam_stns** を入れます。これが§2で見たNSS／PAMの差し込み口に挿さり、「`alice` って誰？」「通していい？」の問い合わせ先を、ローカルファイルの代わりにSTNSサーバー（HTTP）へ向けます。LDAP構成で `ldap`／`pam_ldap` を挿していた場所に、STNS版を挿すだけ、という対応関係です。

**クライアント側にキャッシュを持つ。** HTTPで都度問い合わせると聞くと「中央のSTNSサーバーが落ちたら全滅では？」と不安になりますが、STNSはクライアント側に直近の結果をキャッシュします。だからサーバーが一時的に落ちても、キャッシュのある範囲でログインは耐えます。

STNSが軽量なのは、**汎用の台帳をやめて「Unixのユーザーとグループを配る」という用途に絞った** ためです。組織図やアドレス帳、証明書は扱いません。その代わり、設定はTOML、通信はHTTPという一般的な道具だけで構成されています。

## 7. libnss-stns / pam_stns の仕組み

§6で、libnss-stns を入れるとユーザー情報の問い合わせ先がSTNSサーバー（HTTP）になる、と書きました。ここでは、その内部を見ていきます。`id alice` や `getent passwd alice` のような通常のコマンドが、コードを変更せずにHTTP経由でユーザーを解決できるのは、NSSがプラグイン機構だからです。

glibc は、`nsswitch.conf` に書かれた名前をもとに `libnss_<名前>.so` という共有ライブラリを実行時に dlopen し、その中の決まった名前の関数を呼びます。ユーザー名からアカウント情報を引く場合は `_nss_<名前>_getpwnam_r()` です。`nsswitch.conf` に `stns` と書けば、glibc は `libnss_stns.so` を読み込み、`_nss_stns_getpwnam_r()` を呼びます。

`libnss_stns.so` の実体はHTTPクライアントです。`_nss_stns_getpwnam_r("alice", ...)` が呼ばれると、その中で次の処理を行います。

- `socket()` でソケットを作る
- `connect()` でSTNSサーバーに接続する
- `write()` で `GET /v1/users?name=alice` のようなHTTPリクエストを送る
- `read()` でレスポンス（JSON）を受け取る

得たユーザー情報は、NSSが期待する `struct passwd`（uid・gid・home・shell などを収める構造体）に詰めて返します。呼び出し側のコマンドは、情報がローカルファイル由来かHTTP由来かを区別しません。その差はNSSの層で吸収されます。

PAM側の `pam_stns` も構造は同じで、認証時にPAMから呼ばれ、その中でSTNSサーバーへHTTPで問い合わせます。

この動作は `strace` で確認できます。`getent passwd alice` をトレースすると、`libnss_stns.so` の読み込みと、それに続く `socket` → `connect` → `write` → `read` のシステムコールが観測できます。

## 8. STNSとLDAP、同じ点と違う点

ここまでをまとめます。まず、STNSとLDAPは **同じ骨格** を持っています。どちらも「中央にユーザーの情報源を置き、各サーバーはそこへ問い合わせる」という構造で、その問い合わせは§2のNSS／PAMという同じ差し込み口を通ります。ログインしてくるサーバーが手ぶらである点も同じです。つまり大きな設計思想は共通で、違うのは「差し込み口の先に、何を、どんな流儀で置くか」だけです。

その違いを並べると、こうなります。

| 観点 | LDAP系サーバー（OpenLDAP等） | STNS |
|---|---|---|
| プロトコル | LDAP | HTTP |
| データモデル | 木（DIT）＋スキーマ | TOML（素朴・フラット） |
| 守備範囲 | 汎用ディレクトリ（何でも入る） | Unixユーザー/グループに特化 |
| 運用 | 重い・学習コスト高 | 軽い・学習コスト低 |
| 可用性 | 別途レプリケーション設計 | クライアントキャッシュ前提 |

表のどの行も、結局は「汎用で重いか、特化で軽いか」という一本の軸から派生していることに気づきます。LDAPは汎用の台帳だから、専用プロトコルもスキーマもレプリケーション設計も必要になる。STNSはユーザー配布に特化したから、HTTPとTOMLとキャッシュで済む。もちろんトレードオフはあります——STNSはユーザーとグループ以外を扱えませんし、汎用ディレクトリとしての柔軟さはLDAPにしかありません。どちらが上ということではなく、**要件に対して過不足のないほうを選ぶ**、という話です。

## 9. STNSのその先 — 業界はどこへ向かうか

最後に、そのさらに先です。STNSはLDAPを軽くしましたが、業界の流れはもう一歩進もうとしています。ここは特定の製品や特定の組織の事情の話ではなく、一般的な技術トレンドとして書きます。

そもそもの前提が変わりました。LDAPもSTNSも、根っこにあるのは「各ホストに、同じユーザーを配って揃える」という発想です。ところがクラウドでは、サーバーはオートスケールで増えたり消えたりします。数分だけ生きて消えるホストに、いちいちユーザーを同期し続ける——それ自体が、だんだん割に合わなくなってきました。

そこで、大きく二つの方向が進んでいます。

ひとつは、**認証をクラウドのIdP（Identity Provider）に集約する** 方向です。Okta、Microsoft Entra ID、JumpCloud、Google Workspace といったサービスに社員のアイデンティティを一元化し、OIDC や SAML といった標準プロトコルで各所の認証をそこへ委ねます。「ユーザーの単一の情報源を中央に」という§1の発想はそのまま、その中央がクラウドのIdPになった、と捉えると分かりやすいです。

もうひとつは、**そもそもホストに恒久的なユーザーや固定の鍵を置かない** 方向です。代わりに、ログインが必要になった瞬間に、短時間で失効する使い捨ての証明書やセッションを発行します。SSH証明書を発行する Teleport や HashiCorp Vault、Smallstep、あるいはサーバーへの接続そのものを仲介する AWS Systems Manager Session Manager や Boundary といった道具が、この系統です。

この方向の利点は、たとえば「退職者の鍵がどこかのホストに残り続ける」という事故が、構造的に消えることです。アクセスは中央を必ず通るので監査ログが残り、MFA（多要素認証）とも自然に統合できます。

一言でまとめると、こういう流れです。LDAPからSTNSへの進化が「**中央化を軽くした**」ものだとすれば、その次は「**そもそもホストにユーザーを置かない**」——中央に置くことすらやめて、必要なときだけ一時的な通行証を出す、という方向へ動いています。

## 10. まとめ

最後に、要点をまとめます。

- **NSS／PAMは、差し替え可能な土台。** Linuxが「誰か」「通していいか」を確かめる場所で、ここに何を接続するかで、ユーザー管理の方式が決まります。
- **LDAPは中央化の元祖。** もともとは組織の情報を何でも収める汎用ディレクトリで、強力な代わりに重い。ユーザー管理は、その一用途にすぎませんでした。
- **STNSは同じ土台に乗る特化型の実装。** 汎用をやめてUnixユーザー／グループの配布に絞り、設定はTOML、通信はHTTP。クライアント（libnss-stns / pam_stns）は、NSS/PAMの規約に沿って動くHTTPクライアントです。
- **そして次は、IdP＋短命クレデンシャルへ。** 「中央に置く」から「そもそも置かない」へ。
